# 下载中转站

<p><img src="assets/download-transfer-station.png" width="128" alt="下载中转站图标"></p>

把 IDM 下载完成的视频自动中转到当前 Eagle 资源库的“全部”。捕获到视频所在网页时会保存为 Eagle 来源；没有来源也会直接导入。

## 1.2.10：多视频页面候选恢复

当前版本为 1.2.10。同一个标签页、同一个网站可以同时存在多个独立视频；一个页面解析候选不再把其他完整 MP4、HLS 或 DASH 误判成同一视频并隐藏。只有结构上已经确认是传输分片的资源才进入技术区。

对于 `blob:`/MSE 播放器，扩展会从播放器向外寻找最近的稳定内容链接：作品页主视频保留当前网址，相关推荐或信息流视频分别使用自己的作品链接。该规则按页面结构和内容身份工作，不包含 Pinterest 或其他站点域名分支。

## 1.2.9：结构化视频页面发现

当前版本为 1.2.9。网页明确声明 `og:type=video`、`twitter:card=player`、规范作品地址和播放器地址时，即使播放器使用直链、跨域框架或内部结构无法读取，扩展也会把该规范网页建立为内容级页面解析候选。Vimeo 的 `/746646949` 这类数字作品页不再因为路径缺少 `/video/` 而只剩“未关联播放资源”。

该入口完全依据网页公开的结构化视频元数据和同源规范地址，不按 Vimeo 或其他域名分支，不猜测主页或普通网页。播放器底层 HLS/DASH/分片仍保留在技术诊断区且不可误提交；下载继续由固定 yt-dlp、FFmpeg 和 FFprobe 的统一桌面流程完成。

## 1.2.8：通用页面分轨格式兼容

1.2.8 是历史分轨格式兼容版本。通用页面解析不再假定独立音轨一定使用 `.m4a`：优先选择 MP4 视频轨与 M4A 音轨，同时兼容 Pinterest 等页面返回的“MP4 视频轨 + MP4 音频轨”，再按容器兼容性逐级回退。所有组合仍由固定 yt-dlp 解析、由本机 FFmpeg 无损合并，不新增逐站下载器。

现场 Pinterest Pin `72690981480504869` 只提供五档 H.264 HLS 视频轨和一条扩展名为 MP4 的 AAC 音轨；1.2.7 因固定要求 M4A 而报 `Requested format is not available`。1.2.8 已验证选择 720×900 视频与 AAC 音频并合并为约 12 秒的有效 MP4。

## 1.2.7：通用信息流媒体发现生命周期

1.2.7 是历史信息流发现版本。对于主页、推荐流和瀑布流中的多个 `blob:`/MSE 播放器，扩展以每个视频最近的内容容器和同源永久链接建立内容身份，并从该容器自己的语义标题、说明文字、可读正文、封面和时长生成一张内容卡片，不再用标签页标题描述所有视频。

信息流持续插入/移除 DOM 时，媒体扫描使用有上限的合并调度：第一批变化会在 250 ms 内执行，后续连续变化不能无限推迟扫描。打开或刷新 popup 会主动确认当前网页的发现脚本仍可响应；若扩展曾重载而网页保持打开，则只向当前 HTTP(S) 标签重新注入候选逻辑和内容发现脚本，并立即扫描现有播放器。因此刚滚入视口、正在播放或跨扩展重载保留的内容都不依赖下一条网络请求才出现。

同一标签页已经存在这些可识别内容候选时，无法证明属于哪张内容卡片的 HLS/DASH/音视频网络请求会退出默认列表，只在“显示未关联资源与播放分片”诊断区以紧凑技术项保留且不可提交，防止选中错误视频。没有稳定内容永久链接的网站仍保留其唯一完整清单，不会因降噪丢失下载入口。工具栏角标和展开后的媒体数量统一使用相同的默认可见归组结果。该机制按 DOM 内容容器、永久链接与身份置信度工作，不包含 X 或其他示例网站白名单。

## 1.2.5：抖音逐视频标题与候选身份

1.2.5 是历史抖音逐视频标题版本。抖音精选页会为每个已加载的 `feed-item` 从自己的视频容器读取明确视频 ID、作者和左下角文案，形成独立的 `douyin:<videoId>` 内容候选；当前项与页面预载项不会再共用标签页标题。无法和某个视频 ID/播放器精确绑定的底层网络请求只进入隐藏技术分片，不能继续作为同名可下载视频出现。

## 1.2.4：抖音当前主视频适配

1.2.4 是历史主播放器修复。抖音精选页会同时保留当前播放器和下一条预加载播放器；旧逻辑跳过有 HTTP 地址的当前视频，只处理 blob，因而可能把屏外的 0:49 预加载项描述成正在观看的 3:43 视频。扩展按视口交集、播放状态、播放进度和画面面积选出唯一主播放器；当前播放器有完整 HTTP 流时直接提交该流，没有时才将明确的 `modal_id` 规范化为 yt-dlp 支持的 `/video/<id>` 页面解析候选。

抖音专用层只负责当前内容身份、标题、真实帧、时长、尺寸与规范地址；下载、合并、FFprobe 校验和 Eagle 导入仍完全属于桌面端。直接流和页面解析候选统一使用 `douyin:<videoId>`，不会与推荐项、下一条预加载项或旧导航页混组。桌面同时兼容旧会话中的 `jingxuan?modal_id`，并把不受支持地址、新鲜会话缺失分别显示为可操作错误。

## 1.2.3：通用内容页解析与字节分片重建

1.2.3 是历史通用解析版本。Instagram 等站点会把一个 MP4 拆成带 `bytestart/byteend` 的字节请求；169 KB、288 KB 这类响应只是播放器分片，不能直接交给 FFmpeg。扩展现在通用识别这类参数，保留签名参数并去掉传输区间形成完整地址；Instagram CDN 可公开的资产 ID、时长和码率只用于把同一音视频轨归成一个候选组，不用域名判断下载流程。存在稳定内容页解析候选时，重建地址退到“显示技术分片”区域，默认列表只显示带真实当前帧、标题和时长的内容项。

对于 DOM 只暴露 `blob:`、网络层只暴露分片的网站，扩展从当前视频附近选择同源稳定内容地址，再提交 `resolver=page`；桌面使用随包固定的 yt-dlp/Deno 和当前标签页的最小会话上下文解析最佳完整媒体，随后仍由 FFmpeg 下载、合并、验证和交付。该流程按链接结构覆盖 Instagram 帖子/Reels、TikTok/抖音视频、X 状态、Reddit 帖子、Facebook Watch、Pinterest、Twitch/Vimeo/Dailymotion 等解析器支持页面，不维护“测一个网站加一个下载器”的分支。没有稳定内容地址、解析器不支持、登录过期或 DRM 时明确失败，不猜 URL、不绕过保护。

## 1.2.2：YouTube SABR 全画质

当前版本为 1.2.2。YouTube 现行 SABR 播放响应会公开 1440p、1080p、720p、480p、360p、240p、144p 等真实画质元数据，却不再为自适应格式提供可复用 URL；旧逻辑因此只能保留唯一带直链的 360p progressive 格式。

扩展现在把当前播放器实际公开的分辨率去重后作为一个视频的画质目录，默认推荐最高档。选择画质后只提交页面、分辨率和当前标签页的最小请求上下文；桌面使用固定且校验哈希的 yt-dlp 2026.06.09 与 Deno 2.8.1 解析所选画质的视频轨和最佳音轨，再由 FFmpeg 下载、streamcopy 合并、校验、保存并按选择导入 Eagle。Cookie 不写数据库、不进入命令行或日志，只写入任务专用临时文件并在解析结束的 `finally` 中立即删除。DRM 仍只识别和阻断。

## 1.2.1：YouTube 结构化质量目录

当前版本为 1.2.1。YouTube 使用 MSE 播放时，页面 `<video>` 只暴露 `blob:` 或同源占位地址，普通网络监听也可能只看到几 KB 的探测请求。扩展现在在 YouTube 页面主执行环境读取播放器公开的 `streamingData` 目录，把当前视频的标题、缩略图、时长以及所有带直接地址的自适应音视频格式按 `videoId` 组成一个媒体候选组；每个 itag 独立保留分辨率、帧率、编码、码率和预计大小。

同一个 `/videoplayback` 路径下的不同 itag 不再互相覆盖，带 `range=0-...` 的小探测请求和无身份的小于 128 KB 音视频请求默认归为不可下载分片。用户看到的是当前视频的真实清晰度目录，而不是四条 7 KB“音频”。选择视频和音频后仍统一提交本机软件，由 FFmpeg 下载、streamcopy 合并、校验并按选择交付；扩展没有恢复浏览器下载器。播放器只提供加密签名字串而没有可直接访问 URL 的格式不会被伪造成可下载项，也不实现签名或 DRM 绕过。

## 1.2.0：扩展发现、软件交付

当前版本为 1.2.0。扩展只负责发现媒体、形成内容组、显示真实预览、筛选和提交任务；普通直链、分轨视频、HLS/DASH 的下载、合并、校验、保存与 Eagle 导入全部由本机软件完成。旧浏览器下载/解析页、录制器、移动 UA、密钥面板、在线 FFmpeg、第三方路由及其图片和浏览器库已从活动载荷删除。

不可独立下载的播放分片默认隐藏，筛选器会显示隐藏数量；需要排查时可勾选“显示不可下载的播放分片”，它们只以紧凑行出现且不能提交。媒体按捕获时间从旧到新排列，首次打开自动选中并滚动到最底部；用户正在查看旧内容时不会被新请求抢走选择。

“仅下载”完成后，扩展任务页和桌面软件都可对该 `completed_local` 文件执行“导入现有文件”，直接进入 Eagle 队列而不重新下载。文件路径仍由桌面根据 `planId` 解析并校验在程序的 `已完成` 目录内。

运行时使用下载中转站自有图标。项目发行许可统一为 GPL-3.0；因为仍包含修改后的 GPL 来源文件，固定上游源码和版权记录依法保留为对应源码归档，不作为第二套 UI 或运行时依赖。

## 1.1.12：固定字节 MP4 分片不再冒充完整视频

该版本处理了 Vimeo 等播放器把一段字节区间包装成 `.mp4` URL 的问题；现场两条失败地址分别编码了 `range=9207009-13484176` 与 `range=3402514-6864731`，响应大小恰好等于各自区间长度，因此 FFmpeg 报“Invalid data found when processing input”。它们不是可独立下载的视频。

1.1.12 在浏览器和本机增加双层通用门禁：识别查询参数、路径明文或 base64url 编码的固定字节区间；同组已有 HLS/DASH 时隐藏这些分片，只有分片时阻止创建任务并提示继续播放。本机再次校验，旧扩展或异常提交也不会进入 FFmpeg。规则按 URL 结构和精确区间长度判断，不写死 Behance/Vimeo 域名。

## 1.1.11：同一视频的 CDN 备用链接只显示一次

该版本处理了部分网站为同一视频同时请求多个 CDN/签名地址的问题：与播放器精确匹配的主地址带真实帧和时长，备用地址却没有播放器元数据。旧版把备用地址当成另一条内容，因此出现“两项大小相同、链接不同，一项有图、一项没图”。

1.1.11 增加通用强别名归并：同一页面、同媒体类型下，只有响应 MD5/Digest/强 ETag 相同，或长媒体路径相同且字节数完全一致，才把备用地址并入唯一内容组；只保留信息最完整的下载版本，并共享真实标题、帧、时长和质量。文件大小相同但路径/响应身份不同的媒体仍严格分开，不按抖音或任何示例网站写死规则。

## 1.1.10：每个候选预览自己的下载内容

该版本在 1.1.9 的身份边界上补充候选自身预览：为缺少静态帧的普通视频候选直接加载其自身将提交给本机软件的 `http/https` 直链作为只读预览。切换版本时预览 URL 同步切换，加载失败才回退占位，音频不会冒充视频。

该回退不复用当前页面截图、不按抖音/Behance/Tmall 写白名单，也不改变下载路线；所有远程媒体仍由本机软件执行下载。91 项全量测试覆盖“所选直链就是预览直链”、版本切换和音频排除；冻结运行时、隔离安装器和当前电脑覆盖安装均通过，Chrome 多候选实机复验记录见 1.1.10 验证报告。

## 1.1.9：所见视频与下载 URL 强绑定

1.1.9 只有在媒体请求 URL 与某个播放器真实 `currentSrc/src` 一致时，才允许把该播放器的画面、时长和内容组绑定到候选；比较时忽略会轮换的签名查询参数，但不跨主机或路径猜测。页面后台预加载、隐藏播放器和 MSE/blob 请求不再继承当前可见视频的预览，因此不会再出现“左侧选中 45:13 视频，实际 URL 却是另一个 11:48 视频”。

本机软件同时增加输出身份门禁：普通直链若声明时长与 FFprobe 验证后的最终文件相差超过容差，任务以 `output_duration_mismatch` 失败并阻止导入或交付。新增回归使用现场同类的 2713 秒元数据与 708 秒实际文件证明两层保护；91 项全量测试、冻结运行时、隔离安装器和当前电脑覆盖安装均已通过。Chrome 重载后的同内容实机闭环是剩余人工验收。

## 1.1.8：认证读取统一使用 POST

1.1.8 把健康、任务列表、任务详情和任务预览统一为认证 JSON POST，解决 Chrome 跨源 GET 请求头兼容导致的配对后 401。真实抖音下载随后暴露候选预览与 URL 错配，因此 1.1.8 不作为最终闭环版本，由 1.1.9 接替。

## 1.1.7：扩展状态写入串行化

1.1.7 统一串行化了配对令牌、来源补发队列和最近任务状态，修复旧快照覆盖；实机进一步证明配对 POST 已成功且令牌与服务端一致，随后受认证 GET 仍返回 401 并触发条件清除。该传输兼容问题由 1.1.8 的认证 POST 读取接口接替。

## 1.1.6：配对令牌不会再被旧请求误删

1.1.6 首次保护了迟到 401，并把成功提示置于认证健康检查之后；实机复验继续发现其他并发状态写入仍可覆盖令牌，完整修复由 1.1.7 的统一写入队列完成。

## 1.1.5：下载动作与本机任务保持一致

1.1.5 让“仅下载”和“下载并导入 Eagle”共享同一套提交校验：本机软件未连接、未配对、选择无效或 DRM 阻断时两个按钮都会禁用，不会再出现“下载已开始”但桌面数据库没有任务的假成功。成功提示只在本机返回真实 `planId` 后出现，任务页随后从桌面持久任务恢复进度。

popup 每次连接检查都会主动重试安装器的一次性自动配对。正式安装目录中的扩展可自动消费凭据；从源码仓库加载的开发副本按安全规则不保存本机密钥，令牌丢失后应使用桌面六位码重新配对，不能把密钥写进仓库。排障时应先确认 Chrome 实际加载的扩展路径，避免把开发副本和安装副本当成同一个运行时。

## 1.1.4：任务状态、下载位置与全站视频帧

1.1.4 版本让插件任务页以桌面软件的持久任务为唯一事实源：`completed_local` 与 `imported` 必定显示 100%，同步暂时中断时会明确提示并自动重连，不再把创建时的 0% 当成真实进度。旧配对令牌失效后会用安装器的一次性本机凭据自动恢复；不能恢复时回到“需要配对”，不再永久显示“助手离线”。

“仅下载”结果会显示完整保存路径，并提供“打开所在文件夹”。桌面接口只按任务 ID 打开经校验的 `下载中转站/已完成` 目录，网页不能传入任意本机路径。预览不按 Behance、Tmall 或其他域名写白名单：下载前使用通用 `video` 当前帧/播放器矩形/poster/页面媒体元数据回退；下载完成后再从本机 FFmpeg 生成的实际视频帧回传插件。因 DRM、视频尚未解码、播放器完全不可见或源站阻止画面读取而没有帧时，会保留明确占位，不把站点 Logo 冒充视频画面。

## 1.1.3：每个视频自己的动态质量目录

1.1.3 版本的质量选择器没有固定档位表：它显示“本视频 N 档”，只列当前视频或当前 HLS/DASH 主清单实际公开的分辨率；另一个视频可以是 3 档、5 档、8 档或其他数量。除 Vimeo 结构化播放器目录外，扩展也会在本机浏览器内以 2 MB/4 秒门禁读取可访问的通用 HLS/DASH 清单，从 `RESOLUTION`/`Representation height` 提取质量，失败时保留本机 FFmpeg 自动选择，不猜测不存在的档位。

## 1.1.2：清晰质量选择与自适应清单收敛

1.1.2 版本不再把播放器的 MP4 初始化片段和媒体分片作为几十个“版本”列出；发现完整 HLS/DASH 清单后只展示真实质量档位，并按高度排序、标记推荐项。弹窗把“当前播放质量”和“可下载质量”分开显示，所选清晰度由本机 ffprobe 定位对应节目流，再交给 FFmpeg 下载、合并与校验。1.1.1 的真实帧、单 popup 运行时和所有远程媒体统一交给本机软件的边界保持不变。

扩展默认“先看见，再下载”：页面能直接读取视频像素时抓取真实当前帧；跨域播放器读取受限时，由后台截取当前标签页中播放器的可见矩形，只把压缩后的帧保存在扩展内存。poster/公开内容图是次级回退，网站 favicon 绝不冒充视频封面。同一播放器产生的签名参数轮换 URL 会按稳定播放器身份和媒体路径收敛；HLS/DASH 传输分片不再逐条显示，用户看到的是一个内容组以及组内清晰度、音轨或清单选择。

扩展和桌面软件共享同一份持久任务状态。扩展任务页显示实时进度、阶段、错误、停止和本次运行内重试；桌面软件新增“媒体下载”页，显示标题、来源、已处理大小、进度、输出位置以及下载完成后从视频生成的本机预览图，重新打开窗口不会丢失任务。完整签名 URL 和 Cookie/Authorization 等请求上下文只存在于当前软件进程内存；软件重启后未完成任务会明确要求回来源页重新创建，不会猜测或伪造续传地址。

## 分享给别人

从本项目的 GitHub Releases 下载发行 ZIP，或直接发送项目 `release` 目录里的发行 ZIP。接收者不需要安装 Python：

1. 解压整个 ZIP。
2. 双击 `一键安装.exe`，点击“一键安装”。
3. 安装器自动配置 IDM、创建快捷方式、启动助手，并打开 Chrome 扩展管理页和扩展文件夹。
4. 在 Chrome 开启“开发者模式” > “加载已解压的扩展程序” > 选择已打开的 `chrome-extension` 文件夹。

扩展加载后会使用每台电脑独有的一次性凭据自动配对，不需要输入六位码。由于 Chrome 的安全限制，“加载已解压的扩展程序”必须由用户亲自确认，不能由安装器代点。

安装器为兼容旧版本，程序目录仍使用 `%LOCALAPPDATA%\IDM-Eagle自动导入助手`；对外名称、快捷方式、任务管理器和托盘均显示“下载中转站”。它会备份 IDM 原设置；若发现 IDM 已配置其他杀毒软件，不会强行覆盖，并会给出手动提示。卸载入口位于 `Windows 设置 > 应用 > 已安装的应用` 和开始菜单。

## 日常使用

1. 正常打开 Eagle。
2. 在 Chrome 中使用 IDM 下载视频。
3. 下载结束后自动导入；助手没开时，IDM 接收器会静默启动它。

打开扩展时，当前标签页捕获到的内容会按时间从旧到新显示，默认定位最新项；不可下载播放分片默认隐藏。选择内容后，在右侧查看并修改清晰度、音轨、字幕、文件名和预计动作。点击“下载并导入 Eagle”或“仅下载”后，任务都会交给本机软件下载；B 站分轨资源选择视频和音频后由本机 FFmpeg 下载并无损合并。`blob:` 不是本机可访问的真实地址，界面会要求改选捕获到的直链或 HLS/DASH 清单，不会静默退回浏览器下载。“仅下载”完成后可从任务卡或桌面软件直接补导 Eagle，不会再次下载。

右下角助手图标：左键打开软件，右键可打开任务、网站规则、立即检查下载、检查软件更新或退出。主界面的“媒体下载”页显示插件任务和视频预览，“IDM 导入记录”页保留原有自动导入队列；网站规则列表可新增、启停、控制子域名或删除规则。未列出的网站默认关闭自动来源捕获。

扩展还提供“为下一次下载记录当前网页”和“本次下载不导入”。浏览器来源暂时发送失败时会在本地排队并快速补发；同一网页的重复视频请求会先合并，避免可靠来源被误判为不确定。助手会短暂等待来源，超时后仍然无来源导入。文件未稳定时每 2–3 秒自动重试，最多 20 次；其他临时错误最多 12 次，避免无限循环。

## 一键更新

主界面点击“检查更新”，或右键右下角图标 > “检查软件更新”。助手每天最多静默检查一次；发现新版本后，经用户确认才会下载和安装。更新信息使用独立 RSA 签名，安装包还会核对 SHA-256 和文件大小；新版本启动检查失败时自动恢复旧版本。更新不会删除下载记录、网站规则、Chrome 配对或 Eagle 内容。

0.5.0 及更早版本没有内置更新器，需要手动运行一次当前安装器。覆盖安装 1.2.10 后请在 `chrome://extensions` 对已加载的解压扩展点一次“重新加载”。1.2.7 起，保持打开的普通 HTTP(S) 网页会在下次打开 popup 时自动恢复内容发现，无需额外刷新页面；首次换版仍需由用户确认扩展重载，使新 service worker 生效。安装器会从它拥有的扩展目录清理旧工具页和资源。扩展卡片的“来源/路径”必须指向你打算测试的那一份；仓库开发副本需要六位码配对，安装目录副本使用一次性自动凭据。

## 保护与边界

- 原下载文件始终保留，不移动、不删除、不修改。
- 重复文件按完整内容哈希判断。
- Eagle 关闭时任务持久保存，重新打开后自动继续。
- 不长期保存 Cookie、账号、Authorization、请求头、完整签名媒体 URL 或网页正文；所需 Referer/Origin/User-Agent/Cookie/Authorization 只经认证回环接口进入当前本机任务内存，不写数据库或扩展持久存储。终态后清理；软件重启后不尝试猜测失效地址。
- Widevine、PlayReady、FairPlay、SAMPLE-AES 等 DRM 只检测并阻断，不提供规避或解密。
- 新版不提供 Aria2、MQTT、send2local、m3u8dl、浏览器 FFmpeg、录制、移动 UA、密钥面板、自动下载快捷键或第三方在线下载分流；所有可达远程媒体下载动作由本机软件执行。
- 不修改 Eagle `.library` 内部文件，只调用 Eagle 官方本地 API。
- IDM 不提供网页地址；若浏览器完全没有识别到某次下载，该视频会按“无来源直接导入”处理。

## 环境与源码运行

发行包支持 Windows 10/11，IDM 与 Eagle 用于原自动导入流程；媒体捕获扩展支持 Chrome/Edge，并提供 Firefox 独立清单。发行包不要求 Python，自带 FFmpeg 8.1.2、ffprobe、yt-dlp、Deno 和冻结 Python 运行时。

源码开发模式需要 Python 3.11+，运行时代码仅使用标准库。开发入口、打包和测试见 [DEVELOPMENT.md](DEVELOPMENT.md)，详细验收见 [ACCEPTANCE.md](ACCEPTANCE.md)，安装与回滚见 [docs/INSTALLATION_AND_ROLLBACK.md](docs/INSTALLATION_AND_ROLLBACK.md)。
