# 安装、更新、故障排查与卸载

## 推荐安装（分享版）

1. 把发行 ZIP 完整解压。
2. 双击 `一键安装.exe`，点击“一键安装”。
3. 安装器自动完成：
   - 为兼容旧版本，复制到内部目录 `%LOCALAPPDATA%\IDM-Eagle自动导入助手`；对外统一显示“下载中转站”；
   - 生成每台电脑独有的一次性 Chrome 配对凭据；
   - 备份并配置 IDM 的 `VScannerProgram` 与 `VScannerParameters=[File]`；
   - 创建桌面、开始菜单和 Windows 已安装应用条目；
   - 启动助手，打开 Chrome 扩展管理页和正确扩展目录。
4. Chrome 中开启“开发者模式”，点击“加载已解压的扩展程序”，选择已打开的 `chrome-extension` 文件夹。
5. 扩展首次连接时自动配对并销毁一次性凭据。

Chrome 的最后一次加载确认属于浏览器安全边界，安装器不能代替用户点击。安装器会把扩展目录复制到剪贴板，便于在文件选择窗口粘贴。

## IDM 保护规则

- IDM 对应设置为空：自动配置。
- 已指向旧版 `IdmEagleHook.exe`：升级到新安装位置，并把最初状态按空值保存。
- 已指向其他程序：视为安全软件配置，不覆盖；完成页提示手动处理。
- 原值只备份一次，避免更新时把本助手路径误当成用户原设置。

## 更新与修复

重新运行新版 `一键安装.exe` 即可覆盖程序文件并保留 `%LOCALAPPDATA%\IdmEagleAutoImport` 中的队列、网站规则和配对状态。更新扩展文件后，到 `Chrome > 扩展程序 > 管理扩展程序` 点击一次“重新加载”。

1.2.11 新增桌面代理路由：默认自动跟随 Windows 当前用户系统代理，也可选择始终直连或手动 HTTP/Mixed 端口。数据库仍为 schema 5、扩展协议仍为 1。安装器校验 FFmpeg/ffprobe、yt-dlp/Deno 载荷，健康门要求 `version=1.2.11`、`databaseSchema=5`、`extensionProtocol=1`、`mediaReady=true`、`youtubeResolverReady=true`、`downloadEngine=desktop_ffmpeg`；失败时恢复覆盖前程序目录。

## 正常使用

- 打开扩展：顶部直接显示当前标签页媒体候选；普通资源、HLS/MPD 和 B 站分轨最终都交给本机软件。选择“仅下载”时保存到本地并结束，选择“下载并导入 Eagle”时验证完成后再建立 Eagle job。
- Eagle 开着：IDM 完成视频下载后通常几秒内导入。
- 助手未开：接收器自动静默启动，不要求开机启动项。
- Eagle 没开：记录保持“等待 Eagle”，启动 Eagle 后自动继续；低频等待最长约 1 小时，之后停止并允许手动重试。
- 浏览器来源稍晚：助手最多短暂等待 4 秒；仍无来源则直接导入。
- 来源发送失败：扩展先保存在本地，1 秒、2.5 秒、5 秒快速补发，并保留 30 秒兜底。
- B 站下载项变成 `.htm` 并提示“无法从网站上提取文件”：这是旧浏览器组件路线。覆盖当前 1.2.0 后在 `chrome://extensions` 对“下载中转站”点一次“重新加载”，再刷新原视频页；新任务会显示本机处理阶段，由软件在内存中携带 Referer/Origin/User-Agent 拉取并合并，不再生成两条 Chrome `.m4s` 下载。旧任务和过期签名地址不能直接重试。
- YouTube 只显示 360p：1.2.1 只能使用唯一带直链的 progressive 格式；当前 SABR 的其他 56 个自适应条目只有元数据。覆盖 1.2.2、在 `chrome://extensions` 重新加载扩展并刷新视频页；新版应显示一个当前视频内容组，以及播放器实际提供的 1440p–144p 唯一高度目录。选择后保持该标签页打开到任务进入下载阶段，桌面会用当前请求上下文解析所选画质。若提示 `youtube_resolve_failed`，刷新视频页后重新创建任务；不要导出或长期保存浏览器 Cookie。
- Instagram 显示很多 169 KB/288 KB 项并在 2% 报 `Invalid data`：这些是带 `bytestart/byteend` 的播放区间。覆盖 1.2.3、重载扩展并刷新页面；默认列表应显示带当前视频帧、标题和时长的稳定帖子候选，小区间和重建后备只计入隐藏技术分片。旧失败任务已经丢失完整会话，不能重试，请从原帖子重新创建。
- Instagram、TikTok/抖音、X、Reddit、Facebook 等 blob 页面提示 `page_resolve_failed`：保持来源标签页和登录状态，刷新内容页再创建任务。桌面解析器依赖站点仍被固定 yt-dlp 支持；私密/年龄限制内容需要当前标签页有效 Cookie，DRM、已删除内容、站点改版或解析器暂不支持会明确失败，不会退回浏览器下载或猜测地址。
- 抖音正在播放 3:43，但候选显示 0:49 或任务报 `Unsupported URL`：这是旧版跳过 HTTP 主播放器并选中屏外 blob 预加载项，同时把 `/jingxuan` 导航页交给 yt-dlp。覆盖 1.2.4、在扩展管理页重载并刷新当前抖音页；旧候选会被清除，新候选必须显示当前画面和 3:43。有 HTTP 媒体源时直接下载，只有 blob 才使用规范化 `/video/<id>`；`douyin_session_expired` 时刷新登录页面后重新创建任务。
- 抖音一次出现多个视频但标题全部等于标签页：覆盖 1.2.5、重载扩展并刷新当前页。新版按每个 `feed-item` 的视频 ID、作者和左下角文案建立独立候选；无法绑定视频 ID 的底层 blob/MSE 请求默认隐藏。旧会话缓存不会自动获得正确标题，必须刷新来源页重新捕获。
- 主页/推荐流出现很多同名 HLS，无法知道属于哪个视频：覆盖 1.2.7 并重载扩展。保持打开的 HTTP(S) 来源页会在打开 popup 时自动恢复发现并立即扫描；新版从每个播放器最近内容容器取得永久链接、正文、封面和时长，同页已有这些可识别内容时，无法关联具体卡片的网络清单默认进入“未关联资源”技术区且不可下载。此规则不按 X 或其他站点白名单实现。
- Pinterest 等页面报 `Requested format is not available`，但格式清单实际存在视频和音频：覆盖 1.2.8。新版兼容独立 AAC 音轨被标记为 MP4 的页面，不要重试旧任务；回来源页重新创建任务，使新的解析上下文生效。
- Vimeo 等页面视频正常播放，但扩展显示 0 个媒体和若干未关联资源：覆盖 1.2.9、重载扩展并刷新作品页。新版会使用网页公开的视频类型、Player 卡片和同源规范作品地址形成内容候选；底层请求仍留在技术区。
- 多视频页面只显示一个候选，或新增页面解析候选后列表反而清空：覆盖 1.2.10、重载扩展并刷新来源页。新版不会再按同标签页和同域名隐藏其他完整媒体，并会把相关播放器绑定到最近的稳定作品链接；该规则不按 Pinterest 或其他网站白名单实现。
- Chrome 能播放 Behance 等网站，但任务在 2% 报 `403 Forbidden`：这通常是浏览器走代理、桌面下载进程直连。覆盖 1.2.11 后打开 `下载中转站 > 网络`，保持“自动（推荐）”；如果显示未检测到系统代理，改选“手动代理”并填写代理软件的 HTTP/Mixed 端口，例如 `127.0.0.1:7890`。无需安装 Python/FFmpeg，也不强制开启 TUN。旧任务的签名地址可能已过期，请刷新来源页后重新创建。
- 软件重启后旧的未完成媒体任务提示“下载上下文已过期”：这是预期安全行为。完整签名 URL 和请求头从不落盘，请回来源页重新创建任务。
- popup 顶部错误显示“助手离线”但任务仍能创建：重新加载扩展以启用 `/api/media/health` 连接检查修复；无需重新配对或修改端口。
- 候选仍显示站点 Logo/下载图标：覆盖 1.2.0、重新加载扩展并刷新来源页。可信页面帧只给 URL 与播放器真实来源匹配的候选；同内容备用 CDN 地址会并入带帧的唯一内容，其他普通视频会预览自己的直链，源站拒绝扩展页加载或地址已过期时才保留占位。
- Behance/Vimeo 出现多个小 MP4 且下载在 2% 报 `Invalid data`：这是固定字节 Range 分片。升级到 1.2.0、重载扩展并刷新来源页；分片默认完全从内容列表排除，筛选器只在需要排查时显示紧凑诊断项。旧失败任务无需重试。
- 同一个视频出现一项有图、一项没图：升级到 1.2.0、重载扩展并刷新来源页清掉旧会话候选。新版按响应指纹，或长媒体路径加精确字节数归并 CDN 备用地址，不会只因大小相同就混组。
- 选中一个视频却下载成另一个视频：这是 1.1.8 把当前可见播放器身份回退给后台预加载请求造成的。升级到 1.2.0、重载扩展并刷新来源页以清掉旧会话候选；页面帧只有精确匹配才可绑定，无页面帧候选预览自己的下载 URL，明显时长错配会以 `output_duration_mismatch` 阻止交付。
- 桌面显示已完成而插件仍为 0%/助手离线：先重载 1.2.0 扩展。新版通过认证 POST 同步任务，并在轮询失败时显示同步中断而不是假装 0% 仍是实时状态；不能自动恢复时从设置输入桌面六位配对码。
- 配对 POST 已成功、桌面六位码发生变化，但插件仍提示“尚未配对或配对已失效”：这是 1.1.7 的认证 GET 请求头兼容问题。升级并重载 1.2.0；健康和任务读取使用认证 POST，且保留扩展 Origin 与随机令牌校验。
- 显示“配对完成”后按钮仍禁用，重新打开设置又要求配对：1.1.5 有迟到 401，1.1.6 有旧状态写回，1.1.7 还有认证 GET 请求头问题。升级并重载 1.2.0；配对一次后关闭、重开设置，顶部仍应为“已连接”。
- 点击“仅下载”提示已开始，但 popup 和桌面都没有任务：1.1.4 的未配对动作会产生这一误报。升级并重载 1.2.0；顶部若显示“需要配对”，两个下载按钮都应禁用，不会出现成功提示。完成配对后再次点击，任务必须同时出现在 popup 与桌面。
- “仅下载”不知道保存位置：完成任务卡会显示完整路径，点击“打开所在文件夹”。默认位置是 `%USERPROFILE%\Downloads\下载中转站\已完成`；自定义 `IDM_EAGLE_DOWNLOAD_ROOT` 时使用其下同名目录。之后想放入 Eagle，点击任务卡或桌面“导入现有文件到 Eagle”，不需要重新下载。
- Eagle 已打开但助手仍显示“正在等待 Eagle”：新版助手会在 V2 健康接口不可用时自动回退 Eagle 官方 V1 接口；覆盖安装当前发行包后无需修改 Eagle 设置，状态应在数秒内变为“Eagle 已连接”。
- 文件尚未稳定：每 2–3 秒自动重试，最多 20 次；其他临时错误最多 12 次，均不会无限循环。

## 常见问题

### Chrome 没有自动配对

确认下载中转站与扩展版本一致，并且安装后七天内首次加载。在 `chrome://extensions` 的扩展详情中核对加载路径：正式安装副本位于 `%LOCALAPPDATA%\IDM-Eagle自动导入助手\chrome-extension`，会使用一次性凭据；直接从 Git 仓库加载的开发副本故意不含本机密钥，需使用助手显示的六位配对码。若正式凭据已过期或被消费，同样使用六位码手动配对。

## 软件更新与回退

0.6.0 起可在主界面“检查更新”或右下角菜单“检查软件更新”中更新。客户端先验证 RSA 签名，再核对完整 ZIP 的 SHA-256 和大小。安装时旧程序目录会暂存为本机备份；新版本必须在 20 秒内通过本机健康检查，否则安装器自动恢复旧目录并重新启动旧版。数据目录位于 `%LOCALAPPDATA%\IdmEagleAutoImport`，不参与替换。

### 下载后没有记录

进入 `IDM > 选项 > 下载 > 病毒检查选项 > 编辑`，核对程序为安装目录中的 `IdmEagleHook.exe`，参数必须是 `[File]`。若这里属于杀毒软件，请不要覆盖，需选择其他接入方案。

### 一直等待 Eagle

打开 Eagle 并进入目标资源库，再在助手点击“立即重试（可选）”或在托盘菜单点击“立即检查”。

## 卸载与回滚

从 `Windows 设置 > 应用 > 已安装的应用` 或开始菜单运行“卸载助手”。卸载程序会：

- 先退出助手；
- 仅当 IDM 当前仍指向本助手时恢复安装前值；
- 删除桌面/开始菜单入口和程序目录；
- 保留 `%LOCALAPPDATA%\IdmEagleAutoImport` 中的历史、规则和指纹，方便以后重装。

卸载不会删除下载视频、Eagle 项目或 Chrome 官方 IDM 扩展。Chrome 中加载的个人扩展需在扩展管理页手动移除。

## 1.2.11 媒体运行时、更新与回滚

以下行为属于当前 1.2.11 载荷；历史验证值见独立验证报告：

- 安装载荷包含固定 FFmpeg/ffprobe 8.1.2、yt-dlp 2026.06.09、Deno 2.8.1、Chrome/Edge/Firefox 扩展、许可证、二进制哈希和对应源码；所有解析、主下载和音视频合并都在本机执行，不依赖云服务。
- 后端只解析固定安装目录的 FFmpeg/ffprobe。`/health` 同时报告产品版本、数据库 schema 5、扩展协议 1、`desktop_ffmpeg` 下载引擎和媒体工具就绪状态。
- 数据库通过 `PRAGMA user_version=5` 保存捕获会话、候选组和本机下载计划；升级前使用 SQLite 在线备份，不删除 0.6.0 队列。旧 `component_files` 表在迁移中删除，无法恢复完整 URL 的活动任务标为上下文过期。
- 更新健康检查要求 `version=1.2.11`、`databaseSchema=5`、`extensionProtocol=1`、`mediaReady=true`、`youtubeResolverReady=true`。任一失败都恢复旧程序，保留网站规则、Chrome 配对、代理模式、任务、指纹和 Eagle 内容。
- 覆盖升级只有在新后台健康门、快捷方式和卸载注册全部成功后才轮换新的单次 Chrome bootstrap；故障注入回滚发生在轮换前，旧扩展与旧配对保持一致。
- 程序临时目录与用户数据目录分离。卸载器只删除带安装归属的程序资产和可验证的临时中间文件；最终下载文件、用户下载及手动指定文件不在清理范围。
- 浏览器扩展仍需用户亲自确认安装或商店授权。新增权限必须在安装说明中逐项解释，不用企业策略绕过浏览器确认。
- 1.2.11 发行物统一采用 GPL-3.0；YouTube、通用内容页/信息流绑定、桌面代理路由与抖音当前播放器/逐视频标题适配器为本项目独立实现，yt-dlp/Deno 许可证、哈希与 yt-dlp 对应源码随包提供。安装包继续提供上游版权通知、固定源码快照、构建/安装信息和原项目 MIT 历史通知；旧上游 UI、图标与浏览器库不进入活动扩展。
- `packaging/Test-Installer.ps1` 可在隔离根目录重复验证全新安装、0.6.0 升级、强制失败回退和卸载；`Test-FrozenRuntime.ps1` 验证冻结后台、健康门和 IDM 接收模式。
