# 37 扩展状态串行写入

状态：已实现且自动发行门禁通过，等待 Chrome 1.1.7 实机闭环。

## 问题

1.1.6 已阻止迟到 401 直接误删新令牌，但真实 Chrome 仍在配对后写回空令牌。扩展的令牌、来源补发队列和最近任务字段共存在 `downloadTransferStation` 对象中；多个异步调用会同时读取旧对象、分别合并一个字段，再把整份对象写回，产生经典丢失更新。

## 证据

- 实机最后一枚非空令牌 T8 长度 43，摘要与在线 1.1.6 服务端完全一致，Origin 也一致。
- T8 后紧跟一条空令牌记录；`lastPlanId` 长度和 `lastPlanStatus=queued` 未变化，说明旧状态对象覆盖了新令牌。
- 并发 Node 回归在缺少统一状态队列时失败；这是用户症状的最小确定性复现。

## 实现

- `createStateUpdateQueue(readState, writeState)` 串行执行完整的“读取最新状态 → 计算补丁 → 写入”事务。
- 所有对象补丁继续保留未修改字段；队列失败不会阻塞后续操作。
- 来源事件追加/消费改为函数式补丁，在执行时读取最新数组，避免令牌和事件同时丢失。
- 401 清除改为函数式 compare-and-clear，在队列临界区再次确认当前令牌仍等于被拒绝令牌。

## 验收

- 两个并发状态写入后新令牌必须保留。
- 条件清除面对已经更新的令牌必须成为空操作。
- Chrome/Firefox 都在 bridge 前加载同一认证/状态逻辑模块。
- Chrome 重载 1.1.7 后手动配对一次，设置关闭重开保持连接并完成真实“仅下载”。
