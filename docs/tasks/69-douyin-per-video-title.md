# 69 抖音逐视频标题与候选身份

状态：ready-for-human

## 根因

现场页面同时存在两个完整 `feed-item`：当前视频 `7662692425235828009` 的作者/文案为 `@热话动漫 / 这一集的瑞克…`，下一条 `7653805516250025262` 为 `@木板解说 / 第7集：深度拆解…`。DOM 信息完整，但底层媒体使用 blob/MSE，原始网络 URL 无法与 `video.currentSrc` 精确匹配，后台因而回退到标签页标题，生成多条同名候选。

## 实现

- 每个已加载 `feed-item` 从 `video_<id>`、显式属性或当前 `modal_id` 获取视频 ID。
- 标题只使用该容器内的 `feed-video-nickname` 与 `video-desc`；标签页标题不进入抖音专用候选。
- 当前项与预载项分别形成 `douyin:<videoId>`，共享本机下载/解析状态机。
- 无法证明属于哪个视频 ID 的底层请求标记为 `unboundDouyinMedia`，只在技术分片区审计。

## 验收

- 两个现场 fixture 必须得到两个不同视频 ID 和两个不同标题。
- popup 默认列表不显示无身份网络候选。
- 重载扩展并刷新现场页后，各卡片作者、文案、预览、时长和下载结果属于同一视频。
