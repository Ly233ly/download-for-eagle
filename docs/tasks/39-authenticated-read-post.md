# 39 认证读取统一 POST

状态：已实现，自动发行门禁、真实持久配对和认证任务读取均通过；后续内容身份问题由任务 41 处理。

## 现场

- 手动六位码配对 POST 被 1.1.7 服务接受，配对码随即轮换。
- Chrome 写入的 43 字符新令牌 SHA-256 与服务端当前令牌哈希一致，随后又写入空令牌。
- 同一令牌携带正确扩展 Origin 的 GET 返回 200；缺少 Origin 的 GET 返回 401。
- 结论：配对和状态队列均已工作，失败发生在配对后的受认证 GET 传输边界。

## 实现

- 新增 `eagleBridgeRead(path, payload)`，固定使用 JSON POST。
- 健康、任务列表、任务详情和任务预览全部迁移到认证 POST。
- 服务端增加对应 POST 读取路由；旧 GET 保留兼容。
- 服务端仍验证扩展 Origin 与随机令牌；令牌同时由 POST 请求体提供，未放宽网页 Origin。

## 验收

- 新回归在实现前分别得到服务端 404 和扩展静态断言失败，实现后通过。
- 90 项全量回归通过。
- 当前安装 1.1.8 的认证 POST 健康接口返回 200。
- Chrome 重载 1.1.8 后配对跨设置关闭/重开保持，认证任务读取成功；真实下载暴露独立的媒体身份错配，不属于认证传输范围。
