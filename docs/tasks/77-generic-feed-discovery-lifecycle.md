# 77 通用信息流发现生命周期

状态：实现完成，发行与实机复验由任务 80 跟踪。

## 问题

现场网页当前可见播放器已经具备稳定内容永久链接、独立正文、poster、时长和播放状态，纯候选选择函数也能正确输出该永久链接与标题，但 popup 只显示网络层 HLS/音视频清单，没有内容卡片。这证明 1.2.6 的内容模型正确，漏检位于页面扫描的生命周期。

该现象有两个不依赖站点的触发条件：

1. 全页 MutationObserver 持续产生变化时，尾随防抖会不断清除 250 ms 定时器，使扫描长期不执行。
2. 扩展重载后，已打开网页不会自动重新注入内容脚本；新 service worker 可以继续发现网络请求，却无法从 DOM 建立页面候选。

## 实现

1. `createBoundedScheduler` 只在当前没有待执行任务时创建定时器。连续变化合并为一次扫描，但不能把它无限后移；回调执行后允许下一轮。
2. `discoverPageResolvers` 在内容脚本启动时立即执行，不再等待第一次 DOM 变化。
3. 内容脚本响应 `discoverPageResolvers` 探测消息并立即扫描。
4. popup 每次完整刷新在读取候选快照前请求 `ensureDiscovery`。后台先向当前标签发送探测；只有没有接收端时才通过 `chrome.scripting` 注入候选逻辑和内容脚本，再次触发扫描。
5. 恢复范围只允许明确的 HTTP(S) 标签；受限浏览器页面、已响应页面和其他标签不注入。

## 自动验收

- 100 次连续调度只产生一个待执行回调；执行一次后可以安排下一轮。
- 模拟消息接收端缺失时恰好注入一次，文件严格为两个发现脚本，随后探测成功。
- 已响应的标签不重复注入，非 HTTP(S) 标签拒绝恢复。
- A177–A180 为正式验收标准。
