# 35 配对令牌竞态修复

状态：第一层 401 修复已实现；1.1.6 实机仍发现共享状态写入竞态，由 1.1.7 任务 37 接替完整修复。

## 问题

用户输入六位码后看到“配对完成”，下载按钮却仍提示连接本机软件；重新打开设置又要求配对。Chrome 扩展存储显示 43 字符令牌与空令牌交替，最后为新令牌被清空。

## 诊断证据

- 当前加载扩展 ID 为 `phochakdmmmehfamipieafeiigencilf`，与服务端保存的 Origin 一致。
- 浏览器最近一次非空令牌长度为 43，其 SHA-256 与 SQLite 当前 `extension_token_hash` 相同，证伪令牌内容不一致。
- 配对前发出的无令牌请求可以在配对成功后才返回 401；旧 `eagleBridgeApi` 无条件执行 `token=""`，因此旧响应会覆盖新状态。

## 实现

- 每个请求冻结自己的 `requestToken`；401 后读取最新存储令牌。
- 最新令牌与请求令牌不同时保留最新值并重试；只有被拒绝令牌仍是当前值时才清除。
- 新增无 DOM/无 Chrome 依赖的 `eagle-bridge-auth-logic.js`，Chrome 与 Firefox 都在 bridge 前加载。
- 配对 UI 在 POST 成功后继续验证认证健康接口与 `authState`，两者通过后才显示完成。

## 验收

- Node 回归覆盖“无令牌旧请求 + 新令牌”“旧令牌 + 新令牌”“当前令牌确实被拒绝”和“无可用令牌恢复”四种情况。
- 静态回归验证认证逻辑脚本加载顺序、bridge 实际调用该决策层、成功提示位于健康检查之后。
- Chrome 重载 1.1.6 后手动配对一次，关闭再重开设置仍保持连接；真实“仅下载”任务必须在插件和桌面同时可见并完成。
